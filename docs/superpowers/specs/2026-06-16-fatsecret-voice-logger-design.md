# Голосовой логгер еды для FatSecret — дизайн

- **Дата:** 2026-06-16
- **Статус:** утверждён (брейншторм), готов к плану реализации
- **Тип:** личный проект (один пользователь)

## Обзор и мотивация

Ведение дневника питания в FatSecret вручную (вбивать каждый продукт после
взвешивания) рутинно. Цель — надиктовать в Telegram фразу вида «греча отварная
200г, куриное филе отварное 150г» и получить записи в дневнике питания своего
FatSecret-профиля.

Запись в дневник через FatSecret API возможна методом `food_entry.create`
(он же `food.entry.create.v2`): параметры `food_id`, `serving_id`,
`number_of_units`, `meal`, `date`. Ведение дневника входит в **бесплатный
тариф Basic** (5000 вызовов/день, US-датасет). Запись — лёгкая часть; трудные
части — распарсить фразу в пары `{продукт, граммы}` и сопоставить продукт с
конкретными `food_id` + `serving_id`. Языковую проблему (русские названия в
US/English-базе) и платный NLP FatSecret обходим личной таблицей соответствий,
которая наполняется органично через чат.

### Цели
- Надиктовать приём пищи в Telegram → записи появляются в дневнике FatSecret.
- Минимум рутины: знакомые продукты логируются без лишних действий.
- Полностью на бесплатном тарифе FatSecret Basic.

### Не-цели (YAGNI)
- Мультипользовательность.
- Веб-UI / мобильное приложение.
- Редактирование прошлых записей (кроме отмены последней).
- Аналитика питания (есть в самом FatSecret).
- Платный NLP FatSecret, Whisper / собственный STT.

## Общая форма

Один Python-процесс, **long-polling** Telegram-бот на always-on хосте,
состояние в SQLite. Распознавание речи делает клиент Telegram (встроенный
голосовой набор в поле ввода) — бот всегда получает готовый текст, отдельный
STT-сервис не нужен.

Бот реагирует только на сообщения от owner `chat_id` (из конфига); всё прочее
молча игнорирует — это и есть единственная «авторизация» на стороне бота.

### Поток одного сообщения
```
надиктовал в Telegram → текст → [Parser: LLM] → [Resolver: таблица/поиск]
   → известно и уверенно?  ── да ──→ запись в дневник + сводка + [↩ Отменить]
                           ── нет ─→ уточнение в чате (поиск/граммы/кнопки) → запись
```

## Компоненты

Каждый компонент — отдельная единица с одной задачей и явным интерфейсом.

### Telegram-адаптер (`bot.py`)
Приём апдейтов (текстовые сообщения + callback-запросы от inline-кнопок),
маршрутизация в обработчики, отправка ответов. Замок на owner `chat_id`.
Держит краткоживущее состояние диалога разрешения (какие элементы ждут выбора).

### Parser (`parser.py`)
Один LLM-вызов. Вход: сырой текст + список известных алиасов из таблицы.
Выход: структурированный список элементов
`[{name, grams, meal_hint, confidence}]`. LLM здесь же сопоставляет синонимы с
существующими алиасами («греча отварная» → известный ключ «гречка»), что снимает
самую хрупкую часть — fuzzy-матчинг.

### LLM-провайдер (`llm/`)
Тонкий интерфейс `LLMProvider` с единственным методом разбора текста в
структуру. Две реализации:
- `AnthropicProvider` (дефолт, модель `claude-haiku-4-5` — дёшево, хорош в русском);
- `OpenAIProvider`.

Выбор через конфиг: `LLM_PROVIDER=anthropic|openai` + `LLM_MODEL`. Парсер зависит
только от интерфейса, не от конкретного SDK. Промпт и схема ответа (структурный
JSON) — общие; провайдер отвечает лишь за вызов модели и возврат текста/JSON.

### Resolver (`resolver.py`)
Для каждого элемента находит `(food_id, serving_id, grams_per_serving)`:
1. Лукап имени в таблице алиасов.
2. Если незнакомо — запускает in-chat разрешение: `foods.search` → топ-N
   кандидатов inline-кнопками → по выбору `food.get` → выбор
   грамм-денонимированной порции (где `metric_serving_unit == "g"`,
   `grams_per_serving = metric_serving_amount`) → сохранение алиаса навсегда.
3. Если у продукта нет грамм-порции — спрашиваем серию у пользователя кнопками.

### FatSecret-клиент (`fatsecret_client.py`)
OAuth 1.0a подпись (HMAC-SHA1) и методы: `foods.search`, `food.get`,
`food_entry.create`, `food_entry.delete` (для отмены). Реализация поверх
`pyfatsecret`; если она не покрывает нужные методы с oob-OAuth — тонкая
собственная обёртка на `requests-oauthlib`. Хранит user access token+secret.

### Diary writer (`diary.py`)
Считает `number_of_units = граммы / grams_per_serving`, определяет приём пищи по
локальному времени и дату (сегодня, локальная), вызывает `food_entry.create` на
каждый элемент. Возвращает созданные `entry_id`.

Границы приёмов пищи (по локальному времени, переопределяемы в конфиге):
- 05:00–11:00 → `breakfast`
- 11:00–16:00 → `lunch`
- 16:00–22:00 → `dinner`
- иначе → `other`

### Store (`store.py`)
SQLite. Схема:
```sql
CREATE TABLE aliases (
  alias              TEXT PRIMARY KEY,   -- ключ пользователя, напр. "гречка"
  food_id            TEXT NOT NULL,
  serving_id         TEXT NOT NULL,
  grams_per_serving  REAL NOT NULL,
  food_name          TEXT NOT NULL,      -- человекочитаемое имя из FatSecret
  created_at         TEXT NOT NULL
);

CREATE TABLE log (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  ts            TEXT NOT NULL,
  raw_text      TEXT NOT NULL,
  entry_ids     TEXT NOT NULL            -- JSON-массив созданных food_entry_id
);
```

## Критерий «сомнения» (подтверждать при сомнении)

Авто-запись по элементу, если **одновременно**: продукт есть в таблице, граммы
заданы явно, единственный матч. Спрашиваем подтверждение/уточнение, если:
- продукта нет в таблице → флоу `foods.search`;
- несколько правдоподобных матчей или низкая уверенность LLM;
- не указаны граммы или единица не конвертируется в граммы («ложка», «штука»).

Приём пищи определяется по времени автоматически, показывается в сводке и не
блокирует запись (можно переопределить кнопкой).

## Отмена

Ответ на запись содержит inline-кнопку `[↩ Отменить]`. В таблице `log` хранятся
`entry_id` каждой созданной записи; отмена удаляет их через `food_entry.delete`.

## Разовая настройка авторизации

Скрипт `auth_setup.py` гоняет 3-legged OAuth в режиме out-of-band (PIN):
получает request token, печатает URL авторизации → пользователь подтверждает в
браузере → вставляет PIN/verifier → обмен на access token → сохранение в конфиг.
Подходит для headless-хоста.

## Конфиг и секреты

В `.env`/конфиге на хосте (не в коде, не в git):
- FatSecret consumer key + secret (приложение);
- FatSecret user access token + secret (из `auth_setup.py`);
- LLM-ключ (`ANTHROPIC_API_KEY` или `OPENAI_API_KEY`), `LLM_PROVIDER`, `LLM_MODEL`;
- Telegram bot token;
- owner `chat_id`;
- (опц.) границы приёмов пищи, таймзона.

SQLite-файл — только таблица алиасов и лог.

## Стек

- Python 3.11+
- `python-telegram-bot` (async)
- `pyfatsecret` / `requests-oauthlib` для подписи OAuth 1.0a
- `anthropic` и `openai` SDK для парсинга (выбор по конфигу)
- stdlib `sqlite3`
- Запуск: systemd-юнит или Docker-контейнер; для long-polling достаточно
  `python -m bot`.

## Обработка ошибок

- Протухший/невалидный FatSecret-токен → бот просит переавторизоваться
  (`auth_setup.py`).
- Rate limit FatSecret (5000/день) — с большим запасом; при достижении сообщить.
- Пустой/неудачный парсинг LLM → «переформулируй».
- Пустой `foods.search` → сообщить и предложить другое название.
- Сетевые таймауты → один ретрай, затем отчёт пользователю.
- Чужой отправитель → молча игнорировать.

## Тестирование

- **Parser**: фиксированные транскрипты → ожидаемая структура (LLM замокан).
- **Resolver**: лукап по таблице + флоу поиска с мок-клиентом FatSecret.
- **FatSecret-клиент**: тест OAuth-подписи против эталонного примера FatSecret;
  мок-HTTP для эндпоинтов.
- **Diary writer**: маппинг времени → приём пищи; расчёт `number_of_units`.
- **E2E**: скриптовый апдейт → проверка параметров вызова `food_entry.create`.
